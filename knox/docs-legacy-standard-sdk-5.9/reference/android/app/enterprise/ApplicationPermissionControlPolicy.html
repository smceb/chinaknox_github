Class Overview

This class provides APIs to dynamically revoke and rollback Android permission(s) to applications on the device. Generally, applications request one or more permissions to work properly. The application requested permissions are shown to the user and, after user acceptance, are granted to that application. This action happens during application installation. Once the permission is granted, the application can use the permission throughout its lifetime. For more information on Android permissions, see Android security and Android uses-permission

Removing permission(s) from an application does not disable or uninstall the application. The user can still launch the application and use it. The only difference is that when the application tries to access APIs or features that require the removed permission, a security exception is thrown.

During the application installation process in the current Android security model, the device lists all permissions requested by the application to the user. The user must approve them to install the application. The current approach is a one-step process, meaning that the user allows all permission(s) or none. The user cannot give permissions selectively. For instance, if an application requests two permissions "INTERNET" and "SEND SMS", the user cannot give only the "INTERNET" permission and not the "SEND SMS" permission with the default Android security model. The user must grant both permissions to use that application or give up using that application.

As another example, assume an enterprise administrator installs a productivity application 'A' to all employee devices, and that application "A" uses three permissions, "INTERNET", "SEND SMS", and "WIFI". According to enterprise security rules, employees are not allowed to use WiFi connections on their device. The application helps to improve employee productivity but it violates some other enterprise security policy. According to current Android behavior, either the enterprise must compromise its security policy or give up using that application.

Application permission control APIs help in the above situation by revoking selective permission(s). All these APIs are effective only on third- party applications, which are not part of the system image.

Warning : Applications generally assume that once permissions are granted during installation time, the permissions never change. Applications generally do not handle any security exceptions caused by missing permissions. Since these APIs dynamically remove permissions for applications, a removed permission may lead an application into a force close state, which leads to an unpleasant user experience. Device administrators should be aware of these side effects and use these APIs only when absolutely necessary.

Note: Since MDM 5.6, this class supports only 'normal' permission protection level. Support for permission protection levels 'dangerous', 'signature' and 'signature|System' is deprecated. For protection level of 'dangerous', use applyRuntimePermissions(AppIdentity, List, int) and getRuntimePermissions(String, int) APIs.

public boolean addPackagesToPermissionBlackList (String permission, List<String> pkgNameList)

Since: API Level MDM_3.0

API to revoke a permission for an application or list of applications by adding to permission blacklist.

Parameters
permission Application permission to be added to blacklist. The permission can be a standard Android-defined permission or an application-defined custom permission. List of Android-defined permissions
pkgNameList One or more package names as list
Returns
  • true if the permission is successfully added to blacklist, else false.
Throws
SecurityException If caller does not have required permissions
Usage
An administrator can use this API to add a particular permission to a blacklist for an application or list of applications. Adding a permission to a blacklist does not disable or uninstall the application. If permission is added to a blacklist, the corresponding application's permission is removed from the Android granted permission list. When the application tries to access the removed permission, permission is denied and the Android framework throws a security exception.
An administrator can use the wildcard character "*" instead of specifying one or more package names; in this case, all 3rd party applications that use the given permission will have that permission removed from the Android granted permission list.
The policy is enforced immediately if the given application is already present on the device. In case the application is not installed, the policy is enforced when the application is installed. In addition, the policy is enforced at all times once it is applied, even if the application is uninstalled and reinstalled.

 EnterpriseDeviceManager edm = (EnterpriseDeviceManager) getSystemService
      (EnterpriseDeviceManager.ENTERPRISE_POLICY_SERVICE);
 ApplicationPermissionControlPolicy appControlPolicy = edm.getApplicationPermissionControlPolicy();

 //permission to be added to black list.
 String permission = "android.permission.BLUETOOTH";

 // administrator can  specify one or more or all package name like below 
 // one package Name
 List pkgNameList = "com.sta.test";
 // List of package Names
 List pkgNameList = new ArrayList();   
 Collections.addAll(pkgNameList, "com.sta.test1", "com.sta.test2", "com.sta.test3");  
 // All packages
 List pkgNameList = "*";

 try {
     boolean result = appControlPolicy.addPackagesToPermissionBlackList (permission, pkgNameList);
     if (true == result) {
         //permission added to blacklist for list of application
     } else {
         //permission not added to blacklist for list of application
     }
 } catch (SecurityException e) {
     Log.w(TAG, "SecurityException: " + e);
 }
In the above example, the permission "android.permission.BLUETOOTH" is added to blacklist for the application "com.sta.test". The application receives a security exception if it tries to access Bluetooth-related functions.
In case of application sharing, the user Id is shared with other applications (generally an application shares a user Id to run a group of application on the same process and shares data among them); revoking a permission affects all the applications that are sharing the same user Id. For example, applications A, B, and C share a user Id and use permission "PERMISSION-1". If an administrator adds permission "PERMISSION-1" to a blacklist for A, B, or C, the permission is revoked for all three applications. For more information on shared user Ids, see Android shared user
Permission
The use of this API requires the caller to have the "android.permission.sec.MDM_APP_PERMISSION_MGMT" permission which has a protection level of signature.

API to rollback a permission for an application or list of applications by adding to permission whitelist.

Parameters
permission Application permission to be added to a blacklist. The permission can be a standard Android-defined permission or an application-defined custom permission. List of Android-defined permissions
pkgNameList One or more package names as list

Throws

Usage

An administrator can use this API to add a permission to whitelist for an application or list of applications. Applications with a matching permission and package name are removed from the permission blacklist. By default, all permissions granted by the Android framework for an application are in the whitelist, which means an application can access permission.

Administrators can use the wildcard character "*" instead of specifying one or more package names. In this case all third-party applications that use a given permission are added to white list.
In case an application shares its user Id with other applications (generally an application shares it user Id to run a group of application on the same process and share data among them), the rollback permission affects all applications that share the same user Id. For example, applications A, B, and C share their user Id and use permission "PERMISSION-1". If an administrator removes permission "PERMISSION-1" from the blacklist for A, B, or C, the permission is rolled back for all three applications. For more information on shared user Id, see Android-shared user

      (EnterpriseDeviceManager.ENTERPRISE_POLICY_SERVICE);
 ApplicationPermissionControlPolicy appControlPolicy = edm.getApplicationPermissionControlPolicy();

 String permission = "android.permission.BLUETOOTH";

 // administrator can specify one or more or all package name like below 
 // one package Name
 ListpkgNameList = "com.sta.test";
 // List of package Names

 // All packages
 ListpkgNameList = "*";

 try {
     boolean result = appControlPolicy.addPackagesToPermissionWhiteList (permission, pkgNameList);
     if (true == result) {
         //permission added to whitelist for list of applications
     } else {
         //permission not added to whitelist for list of applications
     }

The whitelist is an exception to the blacklist that is created using addPackagesToPermissionBlackList(String, List). If the permissions of an application match in both the blacklist and whitelist, the whitelist takes priority and the permission is granted for that application. For example, if the permission "android.permission.INTERNET" for application "com.sta.test" is present in both the blacklist and whitelist, the whitelist take priority and the permission is granted for application "com.sta.test".

In case of multiple administrators, the blacklist takes priority if the same permission is present in both the whitelist and blacklist but is set by different administrators. For example, if administrator A sets "android.permission.INTERNET" permission in the blacklist for "com.sta.test" and administrator B sets "android.permission.INTERNET" permission in the whitelist for "com.sta.test", the permission in the blacklist for administrator A has priority.

public boolean clearPackagesFromPermissionBlackList ()

Since: API Level MDM_3.0

API to clear all revoked permissions from permission blacklist.

Returns
  • true if the all the permission are successfully cleared from blacklist, else false.
Throws
SecurityException If caller does not have required permissions
Usage
An administrator can use this API to clear all permissions from the permission blacklist that were previously added using addPackagesToPermissionBlackList(String, List). If an administrator calls clear permission blacklist, all blacklisted permissions for applications are added back to the Android-granted permission list for those applications. This API functionality is the same as removePackagesFromPermissionBlackList(String, List) except that the API removes all permissions from blacklist instead of removing specific ones.

In case of multiple administrators, this API clears only the blacklist set by the calling administrator. For example, administrator A sets one set of permissions in the blacklist and administrator B sets another set of permissions. If administrator A calls this API, all blacklisted permission set by administrator A are cleared but blacklisted permission set by administrator B are still valid and enforced.

public List<String> getApplicationGrantedPermissions (String pkgName)

Since: API Level MDM_4.0

API to get a list of granted permissions for an application.

Parameters
pkgName package name
Returns
  • String list of the granted permissions if it succeeds, else null.
Throws
SecurityException If caller does not have required permissions
Usage
An administrator can use this API to get the list of permissions that are currently granted and active for a particular application.

These permissions are different from the requested permissions, which are available through PackageInfo.requestedPermissions; the later lists all permissions requested by the application via its AndroidManifest.xml. There is no requirement that the system grant all requested permissions. For example, permissions with a protection level of "signature" or "signature" may not be granted to an application based on various criteria (Android Permission Protection Levels).

The list of granted permissions is a subset of PackageInfo.requestedPermissions. If other methods provided in this class are used for a particular application, then the list of granted permissions changes accordingly for that application.

 EnterpriseDeviceManager edm = (EnterpriseDeviceManager) getSystemService
      (EnterpriseDeviceManager.ENTERPRISE_POLICY_SERVICE);
 ApplicationPermissionControlPolicy appControlPolicy = edm.getApplicationPermissionControlPolicy();

 //permission to be added to blacklist.
 String permission = "android.permission.BLUETOOTH";

 // administrator can  specify one or more or all package name like below 
 // one package Name
 ListpkgNameList = "com.sta.test";
 // List of package Names
 List pkgNameList = new ArrayList();   
 Collections.addAll(pkgNameList, "com.sta.test1", "com.sta.test2", "com.sta.test3");  
 // All packages
 ListpkgNameList = "*";

 try {
     boolean result = appControlPolicy.removePackagesFromPermissionWhiteList (permission, pkgNameList);
     if (true == result) {
         //permission removed from white list for list of application
     } else {
         //permission not removed from white list for list of application
     }
 } catch (SecurityException e) {
     Log.w(TAG, "SecurityException: " + e);
 }
In the last example, the permission "android.permission.BLUETOOTH" for application "com.sta.test" is removed from the whitelist; if the same permission is already present in the blacklist, blacklisted policy is enforced and the permission "android.permission.BLUETOOTH" is removed from the Android-granted permission list for application "com.sta.test".
Permission
The use of this API requires the caller to have the "android.permission.sec.MDM_APP_PERMISSION_MGMT" permission which has a protection level of signature.
