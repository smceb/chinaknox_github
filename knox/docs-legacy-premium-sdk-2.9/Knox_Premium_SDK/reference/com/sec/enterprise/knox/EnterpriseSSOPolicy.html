Class Overview

This class provides APIs to setup and manage Single Sign-On (SSO) services inside a container. In order to enable the SSO feature, the administrator needs to first install the sso service. Administrator needs to call below api depending on device's knox version.

SSO Solution Types
SSO_TYPE_CENTRIFY : has been supported since Knox 1.0
SSO_TYPE_SAMSUNG : has been supported since Knox 2.0

If Knox sdk version is less than KNOX_ENTERPRISE_SDK_VERSION_2_0
Administrator needs to call installPackage(String, int, EnterpriseContainerCallback) with valid input parameters to install the service. Package path for SSOService apk is currently set to '/system/preloadedsso/ssoservice.apk_' This path is currently defined in the variable, SSO_SERVICE_PACKAGE_PATH. Once it is installed administrator can then set the customerID, (an unique ID for each enterprise generated by the SSO Vendor to map the request with an Enterprise's Active Directory). Once it is set, the administrator can then add container applications into the whitelist, allowing them to use the SSO service.

If Knox sdk version is greater than or equal to KNOX_ENTERPRISE_SDK_VERSION_2_0
Administrator needs to call installApplication(String, boolean) with valid input parameters to install the service. Only one sso solution could be installed in a container. so administrator should uninstall previous sso solution if he wants to swich sso solution type. Since Knox 2.0 only administrator must be allowed to uninstall the SSO Solution and user must be restricted to uninstall SSO.In Knox 2.0 only one SSO service either Centrify's or Samsung's solution can be installed in a specific container. To install Centrify in container id say 100 ,first administrator checks if Samsung SSO is already installed in container 100, and if it is already installed, uninstalls Samsung SSO in container 100 and then installs Centrify's SSO and vice versa.Package path for Centrify's SSOService apk is currently defined in the variable SSO_SERVICE_PACKAGE_PATH and package path for Samsung' apk is defined in the variable SSO_SAMSUNG_SERVICE_PACKAGE_PATH

However, few models do not have preloaded sso service apk because of the limitation of ROM size. Furthermore, SSO_SAMSUNG_SERVICE_PACKAGE_PATH is deprecated since Knox 2.4 and #SSO_CENTRIFY_SERVICE_PACKAGE_PATH is deprecated since Knox 2.6. Therefore, MDM Agent always has to check if the apk file is present in the device or not. If the file exists, MDM Agent has to install it. If the file does not exist, MDM Agent has to skip installation so that IT-admin can install this apk later.

The method isSSOReady(String) should be called before calling other APIs such as deleteSSOWhiteList(String, String, List), getAppAllowedState(String, String), pushSSOData(String, Bundle), setCustomerInfo(String, String, String) and setSSOWhiteList(String, String, List), to check for service bind status. If isSSOReady(String) returns false, setupSSO(String) should be called before calling the above mentioned APIs.

 KnoxContainerManager mKnoxContainerManager = EnterpriseKnoxManager.getInstance().getKnoxContainerManager(this, containerId);
 EnterpriseSSOPolicy mEnterpriseSSOPolicy = mKnoxContainerManager.getEnterpriseSSOPolicy(CENTRIFY_SSO_PKG))
 if(!mEnterpriseSSOPolicy.isSSOReady(CENTRIFY_SSO_PKG)){
     if(mEnterpriseSSOPolicy.setupSSO(CENTRIFY_SSO_PKG) == EnterpriseSSOPolicy.SSO_RESULT_FAILURE){
         return;
     }        
 }

KNOX 1.0

SSO service should be installed in a container before calling getEnterpriseSSOPolicy(). 

 [How to install]
 // When you create container successfully, containerID will be returned via intent.
 // Use this containerID in below API.
 EnterpriseKnoxManager ekm = EnterpriseKnoxManager.getInstance();
 EnterpriseContainerManager ecm = ekm.getEnterpriseContainerManager(containerID);
 ContainerApplicationPolicy appPolicy = ecm.getContainerApplicationPolicy();
 int installType = ContainerApplicationPolicy.INSTALL_ONLY;
      EnterpriseContainerCallback callback = new EnterpriseContainerCallback() {

          @Override
          public void updateStatus(int status, Bundle data){

              switch(status) {
              case EnterpriseContainerCallback.CONTAINER_PACKAGE_INSTALL_SUCCESS:
                  break;

              case EnterpriseContainerCallback.CONTAINER_PACKAGE_INSTALL_FAILURE:			
                  break;
              }
          }
      };
 try {
     File file = new File(SSO_SERVICE_PACKAGE_PATH);
     if(file.exists()) {
         appPolicy.installPackage(SSO_SERVICE_PACKAGE_PATH, installType, callback);
     }
 } catch (SecurityException e) {
     Log.w(TAG, "SecurityException : " + e);
 }

 // When you create container successfully, containerID will be returned via intent.

 EnterpriseKnoxManager ekm = EnterpriseKnoxManager.getInstance();

KNOX 2.0

Samsung's SSO solution requires setting of company's logo, company's name, Kerberos configuration file (krb5.conf) and SAML Identity Provider URL. All the four items are packed in a bundle and pushed. The key-value pairs are defined as follows:

 try {
     Bundle bundle = new Bundle();
     byte[] customerLogoBytes = null;
     byte[] confDataBytes = null;
     byte[] samlIDPUrl = null;

     ...
     
     StringBuffer MDMRevisedIDPUrl = null;
