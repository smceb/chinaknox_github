Class Overview

This class provides a front end to configure firewall rules on Enterprise Container using iptables.

Be careful when using rules with hostnames as iptables use a DNS to translate hostnames to IP addresses. When using this type of rules make sure that you did not block the DNS server port or DNS server IP, and that you have an Internet connection available. If the DNS is blocked (an Internet connection is not available), the firewall enqueues rules with the hostname and applies them when a valid DNS connection is verified.

The firewall resolves hostnames when the device is powered up and when rules are applied. Be aware that during this interval, the hostname IP address is subject to change.

The Firewall policy can conflict with other features that uses iptables to break firewall functionality. Tethering, the Cisco AnyConnect app, data usage (a feature available on the settings menu of Android Ice Cream Sandwich), and B2C firewall (a feature available on the settings menu of some Samsung devices) are known to conflict with the Firewall policy. Do not use the Firewall policy together with these features.

The firewall must be used with tethering disabled.

Firewall definitions
Local Port - The port on the device. For example, to block port 21 (FTP) on the device from receiving connections, you must block "Local Port" 21.
Remote Port - The port on the server. For example, to block the device from connecting to port 80 (HTTP) on a server, you must block "Remote Port" 80.
Wildcard - A character that represents all possible values for a parameter. For example, to block all ports for IP 1.2.3.4, you must specify * as the port parameter and 1.2.3.4 as the IP parameter.

Firewall Rules Types
Allow rules - Firewall rules that allow the device to send traffic to or receive traffic from an specific network location, according to the rule created. The rules can be specified by IP, port, host name, domain, port range, etc.
Deny rules - Firewall rules that block the device from sending traffic to or receive traffic from a specific network location, according to the rule created. The rules can be specified by IP, port, host name, domain, port range, etc.
Reroute rules - Firewall rules that are used to redirect traffic destined to a target to another destination. A proxy server is an example of an alternate destination.
Redirect exception rules - Firewall rules used to avoid traffic being redirected.
Use case example 1:
An administrator sets up a HTTP proxy but wants some IP addresses of the intranet not to be redirected to the proxy. The administrator can use redirect exception rules to set up IPs that should not be redirected to the proxy.
Use case example 2:
An administrator sets up a reroute rule to redirect all native browser traffic to a proxy but wants some IPs of the intranet not to be redirected. The administrator can use redirect exception rules to set up IPs that should not be redirected to the proxy.
Basic rules - The set of all allow rules, deny rules, reroute rules, and redirect exception rules.
NOTE: When using set* or add* APIs, the rules are not immediately applied to iptables (if they are disabled) but only stored in a database. To enable or disable firewall rules, you must use setIptablesOption(boolean) for basic rules. Once successfully enabled, firewall rules are applied to iptables. If all basic rules are removed or cleaned, the rules are automatically disabled.

public boolean addIptablesAllowRules (List<String> rulesList)

Since: API Level KNOX_1.0

This method is deprecated.
Use addIptablesAllowRules(List).

API to add allow rules in iptables.

Parameters
rulesList
Each node in the list must be in one of the following formats:
"hostname:port;portLocation" or
"hostname:port;portLocation;networkInterface"
hostname - IP address, IP range (e.g., 100.0.0.0-100.0.0.10), and domain name are valid entries for the hostname parameter.
port - Port number and port range (e.g., 8080-8085) are valid entries for the port parameter.
portLocation - Specify if this rule must be applied to remote and/or local port(s); "local" and "remote" strings are valid entries for portLocation parameter.
networkInterface: Must be "wifi", "data", or "*". All other values are invalid.

"*" can be used as a wildcard for the parameters (see the example above).
Returns
  • true if rules were added successfully, else false.
Throws
SecurityException If caller does not have required permissions. Only administrator who created container can enforce policy, if the calling administrator is not an owner of container the API throws SecurityException.
IllegalArgumentException If container does not exist or container creation/removal is in progress.
Usage
An administrator can add allow rules on iptables. Allow rules take precedence over deny rules. Allow rules for a specific application are not supported. 

 List<String> rulesList = new ArrayList<String>();
 // allow IP 1.2.3.4 on all remote and local ports
 rulesList.add("1.2.3.4:*;*");
 // allow connection to servers on port 80
 rulesList.add("*:80;remote");
 // allow the device's port 80
 rulesList.add("*:80;local");
 
 // When you create container successfully, containerID will be returned via intent.
 // Use this containerID in below API.
 EnterpriseKnoxManager ekm = EnterpriseKnoxManager.getInstance();
 try {
     EnterpriseContainerManager containerMgr = ekm.getEnterpriseContainerManager(containerID);
     ContainerFirewallPolicy firewallPolicy = containerMgr.getContainerFirewallPolicy();
     // adding rules (the rules are not enabled yet)
     firewallPolicy.addIptablesAllowRules(rulesList);
     // setIptablesOption(true) enables all basic rules
     // previously added
     firewallPolicy.setIptablesOption(true);
 } catch (SecurityException e) {
     Log.w(TAG, "SecurityException: " + e);
 }

API to add reroute rules on iptables.

Parameters
rulesList Each node in the list must be in one of the following formats:
"hostTarget:portTarget;ipProxy:portProxy" or
"hostTarget:portTarget;ipProxy:portProxy;packageName;networkInterface"
hostTarget - IP address, IP range (e.g., 100.0.0.0-100.0.0.10), and domain name are valid entries for the hostTarget parameter.
portTarget - Port number and port range (e.g., 8080-8085) are valid entries for the portTarget parameter.
ipProxy - IP address is a valid entry for the IP parameter.
portProxy - Port number is a valid entry for the portProxy parameter.
packageName: Must be the name of an application package that has Internet permission. All other values are invalid.
networkInterface: Must be "wifi", "data", or "*". All other values are invalid.

"*" can be used as a wildcard for hostTarget, portTarget, packageName, or networkInterface. If hostTarget and portTarget are both set as "*", the proxy is responsible to resolve DNS accordingly, otherwise webpages will be accessed only through their raw IP addresses.
ipProxy and portProxy do not support the "*" wildcard (see the example above).
