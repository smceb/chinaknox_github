Class Overview

This class provides APIs to control application-related functions and restrictions. The API controls the following functions:

- Installing and uninstalling applications
- Restricting installation and uninstallation of applications
- Disabling and enabling applications
- Querying the current state of an application
- Control application behavior
- Control notifications of applications
- Control widgets

All APIs parameters in this class are case sensitive.

MDM 1.0

Parameters
packageName The application package name to be added to the blacklist
Returns
  • true if the application package name has been successfully added to blacklist, else false.
Throws
SecurityException If caller does not have required permissions
Usage

An administrator can use this API to blacklist third-party applications (applications that are not part of the system image) based on the application package name. An application with a matching application package name cannot be installed. If the application package is already installed, the API does not affect the existing installation. If the application package is uninstalled, the package cannot be installed until the application package name is either removed from the blacklist or added to the whitelist.

Wildcard characters like "com.xyz.*"can be used to specify multiple application package names. A Wildcard character is a character that may be substituted for any of a defined subset of all possible characters. The asterisk character ("*") substitutes for one or more characters, and the question mark character ("?") substitutes a for single character.

Installing via ADB installs package to all users by default in multi user enabled devices. So if administrator has disabled the package installation for any user then for all users installations will be blocked via ADB.

      (EnterpriseDeviceManager.ENTERPRISE_POLICY_SERVICE);

          Log.d(TAG,"  add a application  package name to the application"
               +"Package name has been added to blacklist successfully!");
      }else{
          Log.w(TAG," add a application  package name to the application"

public int backupApplicationData (String pkgName, ParcelFileDescriptor data)

Since: API Level

This method is deprecated.
Since MDM 5.5

API to backup application data.

Parameters
pkgName The package name of the application for which data should be saved.
data The backup data returned from the target application. The file saved by the administrator side will be updated after the operation is successful.
Returns
  • Possible values returned are: BACKUP_APPLICATION_SUCCESS when the operation is successfully completed; BACKUP_APPLICATION_FAILED when the operation failed; BACKUP_APPLICATION_SERVICE_ERROR when enterprise application policy service could not be reached; BACKUP_APPLICATION_BUSY_SERVICE when the service is already performing a backup operation or a restoration operation on the same application.
Throws
SecurityException If caller does not have required permissions
Usage
An administrator can use this API to remotely backup application data. It is up to the administrator to decide where the backup data should be stored. The data may be stored locally in the device or it can be opened a socket in the ParcelFileDescriptor object to save it remotely. This operation includes backing up the application's whole file system trees except for content on the SD card. In order to backup the application data located in the SD card (shared storage) the package name "com.android.sharedstoragebackup" must be provided as the package name parameter, but be aware that it will backup all the data from the shared storage, it means from all applications what can result in a large amount of data. The application target is closed if it is running during the backup process, except if it shares the system uid. During data backup, application data is not sent to Google servers as in a normal Android backup operation; enterprise and Android backup operations always run separately. The operation may take a few seconds to complete, so it is recommended to spawn a thread for this task. Only one backup at a time is allowed. If a backup operation is requested while another is currently running the API returns the error BACKUP_APPLICATION_BUSY_SERVICE. The backup operation is not allowed while a restore operation is already running for the same application package name; in this case, the error BACKUP_APPLICATION_BUSY_SERVICE is returned. An administrator can put a broadcast receiver listening for the intent ACTION_EDM_BACKUP_SERVICE_AVAILABLE when the error BACKUP_APPLICATION_BUSY_SERVICE is received, which indicates that the backup service is available again. The administrator can then retry data backup. The ParcelFileDescriptor provided must point to a valid file otherwise the operation will fail and an exception may occur.

NOTE 1: This API provides sensitive data that should be handled with care by the administrator. Don't ever create a ParcelFileDescriptor with MODE_WORLD_READABLE and MODE_WORLD_WRITEABLE flags since you would be exposing this data for any malicious application in the device.

NOTE 2: Due to Android security limitations, this policy will not work when Device Owner is enabled on the device. 

 // administrator wants to restore application's data
 final String packageName = "com.android.providers.settings";
 final EnterpriseDeviceManager edm = (EnterpriseDeviceManager) getSystemService(EnterpriseDeviceManager.ENTERPRISE_POLICY_SERVICE);
 final ApplicationPolicy mAppPolicy = edm.getApplicationPolicy();
 
 Runnable taskToBackupData = new Runnable() {
     public void run() {
         ParcelFileDescriptor data = null;
         // we use dataTempFile so we don't mess up existing backup data if
         // operation fails
         File dataTempFile = new File(getFilesDir(), packageName + ".temp");
         File dataFile = new File(getFilesDir(), packageName + ".data");
         int ret;
         try {
             data = ParcelFileDescriptor.open(dataTempFile, ParcelFileDescriptor.MODE_READ_WRITE
                     | ParcelFileDescriptor.MODE_CREATE | ParcelFileDescriptor.MODE_TRUNCATE);
             try {
                 ret = mAppPolicy.backupApplicationData(packageName, data);
             } catch (SecurityException e) {
                 Log.w(TAG, "SecurityException: " + e);
             }
             if (ret == ApplicationPolicy.BACKUP_APPLICATION_SUCCESS) {
                 // if operation was successfull, store the data in the permanent
                 // file
                 if (dataFile.exists()) {
                     dataFile.delete();
                 }
                 // at this point, you need to copy from dataTempFile to dataFile
                 // copyFiles(dataTempFile, dataFile);
             } else if (ret == ApplicationPolicy.BACKUP_APPLICATION_BUSY_SERVICE) {
                 // a conflicting operation is already running, try
                 // again later (we cannot predict, but usually a
                 // few minutes will be more
                 // than enough) or
                 // put a broadcast receiver listening for the intent
                 // ACTION_EDM_BACKUP_SERVICE_AVAILABLE to retry
                 // when the resource is available
             } else if (ret == ApplicationPolicy.BACKUP_APPLICATION_SERVICE_ERROR) {
                 // the Enterprise service was not available
             } else if (ret == ApplicationPolicy.BACKUP_APPLICATION_FAILED) {
                 // a generic error occurred between the operations like an
                 // IOException.
             }
             if (dataTempFile.exists()) {
                 dataTempFile.delete();
             }
         } catch (FileNotFoundException e) {
             Log.w(TAG, "Error: file not found");
         } catch (IOException e) {
             Log.w(TAG, "Error: IO exception");
         }
     }
 };
 Thread threadToBackupData = new Thread(taskToBackupData);
 threadToBackupData.start();

      (EnterpriseDeviceManager.ENTERPRISE_POLICY_SERVICE);

 File file = null;
 FileInputStream is = null;
 byte[] bytes = null;
 try {
     file = new File(iconPath);
     is = new FileInputStream(file);
 } catch (Exception e) {
     e.printStackTrace();
 }
 if (null != is) {
     // Get the size of the file
     long length = file.length();
     if (length > Integer.MAX_VALUE) {
         throw new IOException("The file is too big");
     }

     // Create the byte array to hold the data
     bytes = new byte[(int) length];
     // Read in the bytes
     int offset = 0;
     int numRead = 0;
     while (offset < bytes.length
             && (numRead = is.read(bytes, offset, bytes.length - offset)) >= 0) {
         offset += numRead;
     }

     if (offset < bytes.length) {
         throw new IOException("The file is not completely read: "
                 + file.getName());
     }
     // Close the input stream, all file contents are in the bytes
     // variable
     is.close();

Throws

Usage

An administrator can use this API to disassociate a default application from executing some type of task. Only applications previously set by the calling administrator will be disassociated from its task.

If the administrator passes null for Intent (task type) parameter, then the given ComponentName will no longer perfom any task type previously set as default by the calling administrator. In the same way, passing null for ComponentName parameter, the given Intent (task type) will not be associated to any application previously set as default by the calling administrator. Finally, if null is passed for both parameters, all associations previously made by this administrator will be removed.

If the administrator disassociates some default application from executing one of the four types of SMS/MMS Schemes , then all other three Intent task types will be automatically removed from MDM database.

If the administrator disassociates some default application from executing one of http and https Schemes , then both of Intent task types will be automatically removed from MDM database.

     }

public int restoreApplicationData (String pkgName, ParcelFileDescriptor data)

Since: API Level

This method is deprecated.
Since MDM 5.5

API to restore application data and preferences.

Parameters
pkgName The package name of the application for which data should be restored.
data The data to be restored, stored into a file.
Returns
  • The possible values returned are : BACKUP_APPLICATION_SUCCESS when the operation is successfully completed; BACKUP_APPLICATION_FAILED when the operation failed; BACKUP_APPLICATION_SERVICE_ERROR when enterprise application policy service could not be reached; BACKUP_APPLICATION_BUSY_SERVICE when the service is already performing a restoration operation or a backup operation on the same application.
Throws
SecurityException If caller does not have required permissions
Usage
An administrator can use this API to remotely restore application data. The administrator must provide the data to be restored, and the restored data must be the result of a previous backup that the administrator has performed through using backupApplicationData(String, ParcelFileDescriptor). To restore the application data from SD Card (shared storage) the input data must be the result of a backup performed using the package name "com.android.sharedstoragebackup", but be aware that it will restore the whole shared storage data.The application state after a restore operation should be the same as it was at the moment of the backup. An application target that is running during restoration process is closed, except if it shares the system uid. The operation may take a few seconds to complete, so it is recommended to spawn a thread for this task. Only one restoration is allowed at a time; if a restoration operation is requested while another one is currently running, the API returns the error BACKUP_APPLICATION_BUSY_SERVICE. The restoration operation is not allowed while a backup operation is already running for the same application package name; in this case, the error BACKUP_APPLICATION_BUSY_SERVICE is returned. An administrator can put a broadcast receiver listening for the intent ACTION_EDM_BACKUP_SERVICE_AVAILABLE when the error BACKUP_APPLICATION_BUSY_SERVICE is received, which indicates that the backup service is available again. The administrator can then retry data restoration. The ParcelFileDescriptor provided must point to a valid file otherwise the operation will fail and an exception may occur.

NOTE 1: This API provides sensitive data that should be handled with care by the administrator. Don't ever create a ParcelFileDescriptor with MODE_WORLD_READABLE and MODE_WORLD_WRITEABLE flags since you would be exposing this data for any malicious application in the device.

NOTE 2: Due to Android security limitations, this policy will not work when Device Owner is enabled on the device. 

 // administrator wants to restore application's data
 final String packageName = "com.android.providers.settings";
 final EnterpriseDeviceManager edm = (EnterpriseDeviceManager) getSystemService
      (EnterpriseDeviceManager.ENTERPRISE_POLICY_SERVICE);
 final ApplicationPolicy mAppPolicy = edm.getApplicationPolicy();
 
 Runnable taskToRestoreData = new Runnable() {
     public void run() {
         File dataFile = new File(getFilesDir(), packageName + ".data");
         ParcelFileDescriptor data = null;
         try {
             data = ParcelFileDescriptor.open(dataFile, ParcelFileDescriptor.MODE_READ_ONLY
                     | ParcelFileDescriptor.MODE_CREATE);
         } catch (FileNotFoundException e) {
             Log.w(TAG, "Error: file not found");
         }
         int ret;
         try {
             ret = mAppPolicy.restoreApplicationData(packageName, data);
         } catch (SecurityException e) {
             Log.w(TAG, "SecurityException: " + e);
         }
         if (ret == ApplicationPolicy.BACKUP_APPLICATION_SUCCESS) {
             // data was restored
         } else if (ret == ApplicationPolicy.BACKUP_APPLICATION_BUSY_SERVICE) {
             // a conflicting operation is already running, try
             // again later (we cannot predict, but usually a
             // few minutes will be more
             // than enough) or
             // put a broadcast receiver listening for the intent
             // ACTION_EDM_BACKUP_SERVICE_AVAILABLE to retry
             // when the resource is available
         } else if (ret == ApplicationPolicy.BACKUP_APPLICATION_SERVICE_ERROR) {
             // the Enterprise service was not available
         } else if (ret == ApplicationPolicy.BACKUP_APPLICATION_FAILED) {
             // a generic error occurred between the operations like
             // an IOException or data was invalid.
         }
     }
 };
 Thread threadToRestoreData = new Thread(taskToRestoreData);
 threadToRestoreData.start();

Usage

An administrator can use this API to set an application to execute a given type of task, passed as parameter in its Intent format. The most commonly used types of task have been defined in constants: SMS_MMS_TASK, LAUNCHER_TASK, OPEN_URL_TASK, OPEN_PDF_TASK, OPEN_DIALER_TASK, PLAY_AUDIO_TASK, PLAY_VIDEO_TASK, DEVICE_ASSISTANCE_ACTIVITY_TASK, DEVICE_ASSISTANCE_SERVICE_TASK. If it is necessary to build your own type of task in an Intent format you might want to see this link. In case a task type has already been assigned to a ComponentName by other administrator, that task will not be re-associated with the new ComponentName.

Once an Intent is associated with a ComponentName, the Android framework will always try to use that ComponentName to handle such Intent. The administrator may check if a ComponentName can handle the given Intent by using this Android API. If an activity that cannot handle the Intent is passed to this API, the chooser dialog will be normally prompted when that Intent is thrown so that the user itself can choose the best activity to perform such task.

An administrator can use this API to clear the data and cache of an application package to reset the application package to its initial state or free up memory. The application package is wiped silently in the background with no user interaction. 

 EnterpriseDeviceManager edm = (EnterpriseDeviceManager) getSystemService
      (EnterpriseDeviceManager.ENTERPRISE_POLICY_SERVICE);
 ApplicationPolicy appPolicy = edm.getApplicationPolicy();
 try {
     boolean result = appPolicy.wipeApplicationData("com.test.app");
     if (true == result) {
         Log.d(TAG, " Wiping the data of an application package has been successful!");
     } else {
         Log.w(TAG, " Wiping the data of an application package has failed.");
     }

 } catch (SecurityException e) {
     Log.w(TAG, "SecurityException: " + e);
 }

Permission
The use of this API requires the caller to have the "android.permission.sec.MDM_APP_MGMT" permission which has a protection level of signature.
